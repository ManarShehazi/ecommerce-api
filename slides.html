🛒 E-Commerce API

A simplified e-commerce application


FastAPI • React • SQLite • SQLAlchemy



Manar Shehazi

What It Does

  • Register / Login — simple session-token auth
  • Browse Products — 5 items in the catalog
  • Add Credits — virtual currency (no payment flow)
  • Purchase Products — debit balance, create order
  • Order History — view past purchases

Architecture

React Frontend ──HTTP/JSON──▶ FastAPI Backend ──▶ SQLite │ │ api.js ┌───────┼───────┐ (axios + routes/ models.py schemas.py token │ interceptor) dependencies.py (DB session + auth guard)

  • Separation of concerns — routes, models, schemas, deps in separate files
  • Frontend is a thin client — all business logic in the backend
  • SQLAlchemy ORM — swap to PostgreSQL with one config change

Database Models

Users

Products

id INTEGER PK name STRING description STRING price FLOAT

Orders

id INTEGER PK user_id FK → users product_id FK → products price_paid FLOAT ← snapshot! created_at DATETIME

💡 price_paid is a snapshot of the price at purchase time. If the product price changes later, the order history stays accurate.

Authentication Flow

1. User sends POST /auth/register with username + password
2. Server hashes password with bcrypt, generates random token
3. Returns { token, user } — client stores token in localStorage
4. Every subsequent request includes Authorization: Bearer <token>
5. get_current_user dependency looks up user by token → returns User or 401

Simple for this scope. In production → JWT with refresh tokens + expiry (Improvement #1).

Purchase Flow

The most critical endpoint — must be atomic

1. Find the product → 404 if not found
2. Check credits ≥ price → 400 if insufficient
3. Debit balance: user.credits -= product.price
4. Create Order with price_paid snapshot
5. Single commit() — both succeed or both roll back

Atomicity: user never loses credits without getting an order.

Design Decisions

  • Session tokens over JWT — simpler; no expiry/signing logic. Trade-off: DB lookup per request.
  • price_paid on orders — historical accuracy, standard e-commerce pattern.
  • Schemas ≠ Models — models describe DB; schemas describe API contract. Prevents leaking password_hash.
  • Single DB transaction — atomic purchase: debit + order in one commit.
  • bcrypt — intentionally slow hashing, protects against brute-force.
  • FastAPI Depends() — dependency injection for DB sessions and auth guard. Clean, testable.
